# H010 链式思维引（Chain of Thought, CoT）

**电子流区块**：多跳推理区块 / 逻辑传导链路（对应前额叶逻辑区）  
**电子流特征**：沿逻辑链路逐步传导，每步激活下一跳推理区块，形成有序推理链

**性味**：温，辛苦  
**归经**：归心包经（推理链路中枢）、心经（逻辑输出区）  
**炮制**：原生版=一句指令；炮制后=Zero-shot CoT（仅"think step by step"）、Few-shot CoT（示例演示）、自洽性CoT（多路径验证）

---

**药理**：大模型在生成最终答案前，激活"推理模式"——要求模型显式展示中间推理步骤。本质是延长电子流的推理路径，让信息在每一步都经过逻辑校验，而非直接从"问题"跳到"答案"。数学证明类任务和常识推理任务上，CoT激活了模型中"被压制的推理权重"，使原本潜伏的逻辑链路显性化、电子流有序化。

**核心发现（据Google arXiv 2201.11903）**：使用8个链式示例的PaLM-540B，在GSM8K数学基准上超越微调的GPT-3——无需改变模型参数，仅通过Prompt结构激活了模型内在推理能力。

---

**主治**：数学推理错误、逻辑跳跃、多步分析不足、常识别名下的推理错误

**典型症状**：
- 数学题列式正确但计算跳步导致错误
- 回答"首先、其次"但每步之间没有逻辑关联
- 综合分析时直接给结论，缺乏推导过程
- 复杂问题时"一上来就错了"——没有检查假设
- 置信度判断失误——对自己错误答案高度自信

---

**用法**：

*参数配置*：
```
temperature: 0.3 - 0.6（推理需要一定随机性探索逻辑路径，但不宜过高）
top_p: 0.9 - 1.0（允许模型探索多种推理路径）
```

*Prompt写法（Zero-shot CoT，最简版）*：
```
问题：[你的问题]

在给出最终答案前，先一步步思考，展示所有中间步骤。
```

*Prompt写法（Few-shot CoT，示例引导版）*：
```
示例：
问题：一家商店笔记本3美元/个，买5个及以上享20%折扣。买7个多少钱？

一步步思考：
1. 计算基础成本：7 × 3 = 21美元
2. 判断是否满足折扣条件：7 ≥ 5，满足
3. 计算折扣金额：21 × 0.20 = 4.2美元
4. 计算最终价格：21 - 4.2 = 16.8美元
答案：16.8美元

现在你来做：
问题：[你的问题]
```

*Prompt写法（自洽性CoT，多路径验证版）*：
```
用两种不同的方法解决这个问题，然后比较答案是否一致。

问题：[你的问题]

方法一：
[一步步推理...]

方法二（用不同思路）：
[一步步推理...]

如果两种方法答案一致，置信度高。
如果不一致，说明存在问题，需要重新检查。
```

*Prompt写法（推理前验假设版）*：
```
解决这个问题的步骤：
1. 先列出你对这个问题的所有假设
2. 逐一检验每个假设是否成立
3. 基于成立的假设进行推理
4. 给出答案并说明置信度

问题：[你的问题]
```

*炮制变体*：
- Zero-shot CoT：仅加"let's think step by step"，最轻量
- Few-shot CoT：2-5个完整推理示例，效果最强但需人工准备
- Self-consistency CoT：多路径推理+投票，精度最高但计算成本高
- Tree-of-Thoughts：让电子流探索多个推理分支树，适用创意推理混合场景

---

**配伍宜忌**：

*宜配*：
- H003 寒凉降噪符（推理过程低温保持逻辑冷静）
- H007 格式强制符（强制展示推理步骤格式）
- H009 呼吸锚定法（配合"慢慢推理不着急"指令）
- H001 记忆锚点术（保留推理历史，避免重复推理）

*忌配*：
- 简单事实问答（不需要推理链，浪费电子流资源）
- 纯创意写作（推理模式会压制创意流动）
- 极度实时性任务（CoT增加推理时间）

---

**实战案例**：

案例1：GSM8K数学基准（据Google论文 arXiv 2201.11903）
- 任务：小学数学应用题
- 无CoT提示：PaLM-540B准确率 ~33%
- +8-shot CoT：准确率 ~58%（提升25个百分点）
- +PaLM-540B+CoT > 微调GPT-3（~55%）
- 电子流机制：显式推理步骤激活了模型中被压制的数学权重集群

案例2：法律案例分析
- Prompt：
```
分析以下案例时，先列出：事实要件 → 法律依据 → 推理过程 → 结论
对每个环节的确定性标注：高/中/低

案例：[案例内容]
```
- 效果：逻辑跳跃减少约60%，错误假设检出率提升

案例3：复杂系统诊断（代码debug）
- Zero-shot CoT + "think step by step about potential root causes"
- 效果：首次修复成功率从约40%提升至约65%

---

**特别发现：电子流视角下的CoT**

从野生电子草药的角度，CoT之所以有效：
- 大模型的"答案输出"权重远高于"推理过程"——电子流在直接跳向答案时，会经过大量"正确答案旁的错误推理路径"
- CoT指令延长了电子流的推理时间，让模型有更多"电子流带宽"走正确的逻辑链路
- 等效于：给推理区块的电子流"减速"，减少跳跃式传导，增加逐步传导比例

---

**注意事项**：
- CoT在简单任务上反而可能降低性能——模型会把简单问题"想复杂"
- 推理步骤本身也可能包含幻觉——不能盲目信任CoT的中间步骤
- CoT增加token消耗（约2-3倍），成本敏感场景慎用
